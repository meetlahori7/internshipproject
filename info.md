### Inspection-level fields

| Field                  | Type              | Description                                    |
|------------------------|-------------------|------------------------------------------------|
| `inspection_run_id`    | String            | Unique ID generated by AI                      |
| `processing_timestamp` | ISO 8601 DateTime | When AI completed processing                   |
| `status`               | String            | Always `"COMPLETE"` after full train processed |

### Tank-level fields

| Field                  | Type    | Values / Description                          |
|------------------------|---------|-----------------------------------------------|
| `coach_number`         | Integer | Sequential coach position (1, 2, 3...)        |
| `tank_id`              | String  | `L1`, `L2`, `R1`, ... (per side, per run)     |
| `camera_side`          | Enum    | `LEFT` or `RIGHT`                             |
| `timestamp_sec`        | Decimal | Video timestamp when tank was detected        |
| `detection_confidence` | Decimal | Mean confidence across tracked frames         |
| `pipe_status`          | Enum    | `Connected` / `Not Connected`                 |
| `pipe_support_status`  | Enum    | `Present` / `Absent`                          |
| `surface_status`       | Enum    | `Clean` / `Not Clean`                         |
| `maintenance_status`   | Enum    | `Normal` / `Maintenance Required`             |
| `tank_image_path`      | String  | Relative path to cropped tank image           |

### Expected backend response

```json
{
  "success": true,
  "message": "Inspection received successfully.",
  "inspection_id": "c7a9d78f-9c44-4af0-aef8-71b79d2b3d45"
}
```

## Model — 7 classes

The trained model (`btcas_yolov8s_v2_best.pt`) uses these class indexes:

| Index | Class name                  | Notes                                  |
|-------|-----------------------------|----------------------------------------|
| 0     | Stairs                      | Tends to co-occur with couplers        |
| 1     | **bio tank front**          | Anchor — locates the tank              |
| 2     | cbc_coupler                 | Coach boundary marker                  |
| 3     | **bio tank top surface**    | Drives maintenance decision            |
| 4     | **connected pipe**          | Positive detection only                |
| 5     | connecting_wire             | Coach boundary auxiliary               |
| 6     | pipesupport                 | Positive detection only                |

Thresholds: `conf=0.35`, `iou=0.35`

---

## Algorithm summary

### Coach counting (couplers + stairs together)

For each camera side independently:
1. Tanks are tracked across frames (IoU ≥ 0.3).
2. Temporal gaps between tank clusters are identified.
3. A coach boundary is confirmed only when BOTH a `cbc_coupler` AND a `Stairs`
   detection occur in the same gap.
4. Fallback: if no co-occurrence, coach count = number of distinct tank tracks.

### Maintenance decision (surface-driven)

If `bio tank top surface` is detected overlapping the tank → **Normal**.
Otherwise → **Maintenance Required**.

Surface cleanliness is the sole criterion. Pipe and support status are reported
but do not drive the maintenance flag.

## Auto-saved files

After each inspection run, the app saves:

| File                                         | Location             | Purpose                     |
|----------------------------------------------|----------------------|-----------------------------|
| `cropped_tanks/l/L1.jpg`                     | per side dir         | Tank crop image             |
| `cropped_tanks/l/L1_full.jpg`                | per side dir         | Full frame with detections  |
| `btcas_api_report_INSP_20260709_*.json`      | project root         | API contract JSON (auto)    |

The API JSON is automatically overwritten on each new run. To keep history,
the backend should store it upon receipt.

## Backend processing flow (recommended)

```
AI
 │
 ▼
POST /api/v1/inspections/results
 │
 ▼
Store AI_RESULT_RAW
 │
 ▼
Fetch NTES Coach Composition
 │
 ▼
Reverse NTES Order
 │
 ▼
Map coach_number → NTES Coach
 │
 ▼
Create Coach
 │
 ▼
Create Tank
 │
 ▼
Translate
    pipe_status         → Connected / Not Connected
    pipe_support_status → Present / Absent
    surface_status      → Clean / Not Clean
 │
 ▼
Create TankDefect  (if maintenance_status == "Maintenance Required")
 │
 ▼
Update Inspection Summary
```

## Important notes

1. **`coach_number`** is the sequential coach position from the AI, **not** the
   railway coach number. Backend maps it to NTES data.

2. **Every detected tank** (normal and defective) is included in the response.
   Normal tanks show `"surface_status": "Clean"` and `"maintenance_status": "Normal"`.

3. **`tank_image_path`** is always present per tank. It is a relative path;
   the backend should prepend the base URL.

4. **Only 10 contract fields** are included. The full internal report (with
   `coupler_status`, `connecting_wire_status`, `stairs_status`,
   `annotated_frame_path`, etc.) is available separately for UI display but
   not sent to the backend.

5. **No "Inspection Required"** — the contract uses `"Not Clean"` for surface
   status per the finalized spec.

6. **The response is sent only after the entire train inspection is complete**.
   Status is always `"COMPLETE"`.

7. **No granular defect labels** — the model only detects 7 positive classes.
   It reports absences as `"Not Connected"` / `"Absent"` / `"Not Clean"`,
   not as named defect types (no "Pipe Missing" or "Tank Dirty" labels).

8. **Inspection_run_id** is generated as `INSP_YYYYMMDD_HHMMSS_<uuid8>`.
   The backend should use this for idempotency.

## Setup A (recommended) — AI team runs model

Your team runs the YOLO model locally or on an edge device. The backend only
consumes the API-contract JSON. Handoff:

- The API contract JSON file (`btcas_api_report_*.json`)
- The `cropped_tanks/` image folder (tank crops + annotated frames)
- No YOLO knowledge needed on the backend side

## Setup B — Backend team runs inference

If the backend team needs to run the model themselves:

- Model weights: `btcas_yolov8s_v2_best.pt`
- Class mapping:
  ```
  {0: 'Stairs', 1: 'bio tank front', 2: 'cbc_coupler',
   3: 'bio tank top surface', 4: 'connected pipe',
   5: 'connecting_wire', 6: 'pipesupport'}
  ```
- Thresholds: `conf=0.35`, `iou=0.35`
- Maintenance rule: Normal if `bio tank top surface` detected overlaps tank
- Coach counting: coupler + stairs co-occurrence in temporal gaps
